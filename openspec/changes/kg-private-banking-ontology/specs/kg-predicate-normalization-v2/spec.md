## ADDED Requirements

### Requirement: Three-pass normalization with strict rejection
The `normalize_predicate(raw_predicate, family_hint="")` function SHALL normalize predicates using three passes: (1) exact match within the hinted family, (2) priority-ordered cross-family search (ownership > investment > role > deal_network > life_event > affinity > location), (3) fuzzy suffix stripping (_of, _by, _with, _in, _for, _at, _to, _from). If all three passes fail, the function SHALL return `None` (not a default family).

#### Scenario: Direct match in hinted family
- **WHEN** `normalize_predicate("owns", family_hint="ownership")` is called
- **THEN** it returns `("owns", "ownership")`

#### Scenario: Variant match in hinted family
- **WHEN** `normalize_predicate("buys", family_hint="ownership")` is called
- **THEN** it returns `("owns", "ownership")`

#### Scenario: Cross-family match when hint is wrong
- **WHEN** `normalize_predicate("invested_in", family_hint="partnership")` is called
- **THEN** it returns `("invested_in", "investment")` (found via cross-family search)

#### Scenario: Cross-family match when no hint provided
- **WHEN** `normalize_predicate("ceo_of")` is called
- **THEN** it returns `("ceo_of", "role")`

#### Scenario: Fuzzy suffix stripping
- **WHEN** `normalize_predicate("invested_in_by")` is called (the `_by` suffix is stripped to find `invested_in`)
- **THEN** it returns a valid canonical form via the fuzzy pass

#### Scenario: Unknown predicate returns None
- **WHEN** `normalize_predicate("random_unknown_verb")` is called
- **THEN** it returns `None`

#### Scenario: "buys a team" and "owns a team" normalize to same predicate
- **WHEN** both `normalize_predicate("buys")` and `normalize_predicate("owns")` are called
- **THEN** both return `("owns", "ownership")`

#### Scenario: "purchased" and "acquired" normalize to same predicate
- **WHEN** both `normalize_predicate("purchased")` and `normalize_predicate("acquired")` are called
- **THEN** both return `("owns", "ownership")`

### Requirement: Priority-ordered family search
The cross-family search SHALL check families in this exact priority order: ownership, investment, role, deal_network, life_event, affinity, location. Ownership is first because it is the most important family for sports deal origination and the most common source of variant explosion.

#### Scenario: Ownership family takes priority
- **WHEN** a predicate could match in both ownership and another family
- **THEN** the ownership match is returned (ownership is checked first)

### Requirement: Input normalization
The function SHALL normalize the raw predicate string by: lowercasing, stripping whitespace, replacing spaces with underscores, and replacing hyphens with underscores before attempting any match.

#### Scenario: Mixed case and spaces are normalized
- **WHEN** `normalize_predicate("Invested In")` is called
- **THEN** it matches `invested_in` and returns `("invested_in", "investment")`

#### Scenario: Hyphens are normalized
- **WHEN** `normalize_predicate("board-member-of")` is called
- **THEN** it matches `board_member_of` and returns `("board_member_of", "role")`

### Requirement: Compatibility shim for predicate_normalization.py
The existing `predicate_normalization.py` module SHALL be converted to a thin wrapper that delegates to `kg_ontology.normalize_predicate()`. It SHALL preserve the old return contract: always returns `tuple[str, str]` (never `None`). When the ontology returns `None`, the shim SHALL return `(predicate, predicate_family or "deal_network")` as fallback. The old `CANONICAL_PREDICATES` dict SHALL be retained for backward compatibility.

#### Scenario: Old import path still works
- **WHEN** code imports `from app.predicate_normalization import normalize_predicate`
- **THEN** the import succeeds and the function works

#### Scenario: Old return contract preserved
- **WHEN** `predicate_normalization.normalize_predicate("random_unknown", "ownership")` is called
- **THEN** it returns `("random_unknown", "ownership")` (never None)

#### Scenario: Known predicate delegates to ontology
- **WHEN** `predicate_normalization.normalize_predicate("buys", "ownership")` is called
- **THEN** it returns `("owns", "ownership")` (same result as kg_ontology)

### Requirement: Updated extraction prompt suppresses noise predicates
The extraction prompt generated by `get_lm_prompt_section()` SHALL list only HIGH and MEDIUM signal predicates. LOW-signal predicates (coaches, competes_with, attended_event, named_in_honor_of) SHALL NOT appear in the prompt, suppressing LLM generation of noise at the source.

#### Scenario: Prompt does not contain LOW-signal predicates
- **WHEN** `get_canonical_predicates_prompt()` generates the prompt section
- **THEN** the string does not contain "coaches", "competes_with", "attended_event", or "named_in_honor_of"

#### Scenario: Prompt contains HIGH-signal predicates
- **WHEN** `get_canonical_predicates_prompt()` generates the prompt section
- **THEN** the string contains "owns", "invested_in", "ceo_of", "partnered_on_deal", "married", "exited"

### Requirement: Updated extraction prompt constrains entity types
The extraction prompt SHALL include a section listing all 10 allowed entity types with instructions to skip entities that do not fit any type. The old entity type list (`person|company|sports_team|location|product|other`) SHALL be replaced with the new 10-type list.

#### Scenario: Prompt lists new entity types
- **WHEN** `get_entity_types_prompt()` generates the prompt section
- **THEN** the string contains "pe_fund", "sports_foundation", "transaction_event", "life_event", "media_rights_deal" and does NOT contain "product" or "other"

### Requirement: Updated extraction prompt uses new family names
The extraction prompt's JSON schema SHALL list the 7 new family names (`ownership|investment|role|deal_network|affinity|life_event|location`) instead of the old 5 (`ownership|employment|transaction|location|partnership`).

#### Scenario: Prompt JSON schema uses new families
- **WHEN** the extraction prompt is generated in `extract_entities_and_relationships()`
- **THEN** the predicate_family enum in the JSON schema lists the 7 new family names

### Requirement: Symmetric relationship detection uses ontology
The `_relationship_already_exists()` function SHALL determine symmetry by checking `predicate_spec.symmetric` from the ontology rather than using a hardcoded `symmetric_families = {"partnership"}` set. This correctly handles per-predicate symmetry (e.g., `co_owns` is symmetric but `owns` is not, both in the `ownership` family).

#### Scenario: co_owns checked in both directions
- **WHEN** checking if `A co_owns B` already exists
- **THEN** the function also checks for `B co_owns A` (because `co_owns.symmetric = True`)

#### Scenario: owns checked in one direction only
- **WHEN** checking if `A owns B` already exists
- **THEN** the function does NOT check for `B owns A` (because `owns.symmetric = False`)

### Requirement: Invalid entity types rejected at extraction time
The `_process_message()` function SHALL skip entities whose type is not in `ALLOWED_ENTITY_TYPE_NAMES` before creating them in the database. A debug log SHALL be emitted for each skipped entity.

#### Scenario: Entity with type "other" is skipped
- **WHEN** the LLM returns an entity with `type: "other"`
- **THEN** the entity is not created in the database and a debug log is emitted

#### Scenario: Entity with type "person" is created
- **WHEN** the LLM returns an entity with `type: "person"`
- **THEN** the entity is created in the database normally
