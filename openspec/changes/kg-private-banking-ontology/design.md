## Context

The current KG extraction pipeline has three structural problems:

1. **Entity type explosion**: The LLM can return any string as `entity_type` (canonically `person|company|sports_team|location|product|other`). The `other` catch-all accumulates noise. `product` has no place in a sports-deal KG. There is no enforcement at ingestion time.

2. **Predicate drift**: Despite `predicate_normalization.py` mapping ~80 variants to canonical forms, the fallback behavior (`return predicate, "partnership"`) silently admits unknown predicates into the KG. Over time the graph accumulates semantically duplicate edges ("owns" vs "buys" vs "purchased") that were not caught by the variant map.

3. **No relevance filtering**: Every extracted relationship is stored regardless of its value to the private banking use case. Coaching assignments, event attendance, and on-field rivalries clutter the graph alongside deal-critical ownership and investment signals.

**Current architecture:**
- `backend/app/predicate_normalization.py` — flat variant→canonical dict, 5 families, lenient fallback
- `backend/worker/entity_extraction.py` — LLM prompt lists all predicates equally, no relevance gate
- `backend/app/db/knowledge_graph.py` — symmetric check hardcoded to `{"partnership"}`, no entity type validation

**Stakeholders:** Private banking relationship managers focused on sports deal origination.

## Goals / Non-Goals

**Goals:**
- A single-module ontology (`kg_ontology.py`) that is the source of truth for entity types, predicate families, and relevance rules
- Hard limits that prevent unbounded growth (10 entity types, 8 predicates per family, confidence floors)
- Three-tier signal classification (HIGH/MEDIUM/LOW) that filters noise before storage
- Strict normalization where unknown predicates are rejected (not silently admitted)
- Domain-specific families (investment, deal_network, affinity, life_event) that model what private bankers actually query
- Zero-downtime migration path from old 5-family schema to new 7-family schema

**Non-Goals:**
- Retroactive re-extraction of all existing KG data (migration helper provided but bulk re-run is a separate task)
- Frontend changes (entity types and predicates are rendered dynamically from the data)
- Database schema changes (entity_type and predicate are already free-text columns)
- Auto-deletion of stale relationships (staleness flagging only; pruning requires human confirmation)
- Per-user or per-team ontology customization (single ontology for all teams)

## Decisions

### D1: Single ontology module vs configuration file

**Decision:** Single Python module (`kg_ontology.py`) with dataclasses and frozen constants.

**Rationale:** A Python module gives import-time validation (assert on type count, assert on predicate count), IDE autocompletion, and type safety. A YAML/JSON config file would require a loader, schema validation, and lose compile-time guarantees.

**Alternative considered:** YAML config loaded at startup. Rejected because the ontology changes infrequently (it is a domain design choice, not a runtime setting) and Python gives stronger guarantees.

### D2: Return None for unknown predicates vs fallback to default family

**Decision:** `normalize_predicate()` returns `None` when a predicate cannot be mapped. The extraction worker drops the relationship.

**Rationale:** The old fallback (`return predicate, "partnership"`) silently polluted the graph. Explicit rejection forces data quality at the boundary. The LLM prompt is updated to only show valid predicates, so `None` returns should be rare.

**Alternative considered:** Log a warning and store with a `_unclassified` family. Rejected because it creates a new unbounded growth vector.

### D3: 7 families replacing 5

**Decision:** Split `employment` → `role`, `transaction` → `investment`, `partnership` → `deal_network`. Add `affinity` and `life_event`.

**Rationale:**
- `role` is more precise than `employment` (we care about decision-maker identification, not generic employment)
- `investment` captures LP/GP/exit dynamics that don't fit the old `transaction` catch-all
- `deal_network` explicitly models who-does-business-with-whom (the core banking query)
- `affinity` captures sports fandom signals that support relationship building
- `life_event` captures wealth triggers (marriage, divorce, inheritance)

### D4: Only show HIGH+MEDIUM predicates in LLM prompt

**Decision:** The extraction prompt generated by `get_canonical_predicates_prompt()` omits LOW-signal predicates entirely.

**Rationale:** If the LLM never sees `coaches` or `competes_with` in the allowed list, it won't generate them. This is upstream prevention rather than downstream filtering.

### D5: Entity relevance scoring is advisory, not blocking

**Decision:** `should_keep_entity()` is called after extraction and logged, but entities are not auto-deleted. Only relationships are hard-filtered.

**Rationale:** Deleting entities that might gain HIGH-signal relationships from a future extraction would cause re-creation churn. The scoring identifies pruning candidates for periodic review.

### D6: Compatibility shim for predicate_normalization.py

**Decision:** Keep `predicate_normalization.py` as a thin wrapper delegating to `kg_ontology`. Old return contract preserved (always returns tuple, never None).

**Rationale:** Other modules may import from `predicate_normalization`. A shim avoids a shotgun surgery refactor while directing new code to `kg_ontology`.

## Risks / Trade-offs

- **[Risk] Existing KG edges use old family names** → Mitigation: `translate_legacy_predicate()` maps old→new. Migration can run as a batch job or on-read.
- **[Risk] LLM may still generate unlisted predicates despite prompt constraints** → Mitigation: `normalize_predicate()` returns None for these; the extraction worker drops them. Counters track how often this happens.
- **[Risk] Removing `other` entity type drops entities the LLM can't classify** → Mitigation: The prompt instructs the LLM to skip unclassifiable entities. False-negative rate is acceptable because the KG is intentionally curated, not exhaustive.
- **[Risk] 8-predicate-per-family cap may be too restrictive for future needs** → Mitigation: The cap is enforced by assertion. Adding a 9th predicate requires explicit retirement of a less-useful one, which is the desired design discipline.
- **[Trade-off] Dropping LOW-signal relationships loses some context** → Accepted: A relationship like "competes_with" between two sports teams is publicly known and adds no private-banking value. The KG is not a general-purpose encyclopedia.

## Migration Plan

1. Deploy `kg_ontology.py` alongside existing `predicate_normalization.py`
2. Update `entity_extraction.py` to import from `kg_ontology` (new prompt, new normalization, relevance filtering)
3. Convert `predicate_normalization.py` to compatibility shim
4. Update symmetric relationship detection in `_relationship_already_exists()`
5. Verify with unit tests (ontology module + extraction pipeline)
6. Existing KG data continues to work — old family names are valid free-text in the DB
7. Optional follow-up: batch migration of legacy edges via `translate_legacy_predicate()`

**Rollback:** Revert `entity_extraction.py` imports to `predicate_normalization` (the shim still works). No DB changes to undo.

## Open Questions

- Should the `max_entities_per_type_per_team` limit (500) be enforced at ingestion time or only during periodic compaction?
- Should relationship staleness flagging (730 days) be a background job or a manual review process?
- When a person is both a `ceo_of` a company AND `owns` a team through that company, should the KG infer a transitive ownership relationship or leave that to query-time traversal?
